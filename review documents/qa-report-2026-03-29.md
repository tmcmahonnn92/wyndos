### F-01 — Google OAuth Config Error Text Exposed to End Users
**Severity:** High  
**Area:** Authentication — Sign-In Page  
**Category:** Security / UX  

**Steps to Reproduce:**
1. Visit `https://wyndos.io/auth/signin` or `https://wyndos.io/auth/signup` without signing in.

**Expected Result:**
The Google sign-in button should either not appear, or show "Google sign-in unavailable."

**Actual Result:**
The page renders the text: *"Google sign-in needs AUTH_GOOGLE_ID and AUTH_GOOGLE_SECRET"* as live visible UI content.

**Risk / Impact:**
- Discloses internal environment variable names (`AUTH_GOOGLE_ID`, `AUTH_GOOGLE_SECRET`) to every visitor — a minor but real information disclosure risk (OWASP A05).
- Highly unprofessional for any user who sees the sign-in form. It creates an impression of a broken or unfinished product.
- This text is visible on both the sign-in AND sign-up pages.

**Root Cause:**
The conditional in `src/auth.ts` determines `googleEnabled = Boolean(process.env.AUTH_GOOGLE_ID && process.env.AUTH_GOOGLE_SECRET)`. A corresponding UI component renders this raw error string when `googleEnabled` is false rather than hiding or gracefully disabling the button.

**Suggested Fix:**
In the Google sign-in button component, when `googleEnabled` is false: completely omit the Google button from the DOM. Do not render any visible text mentioning environment variable names. The error message (if kept at all) belongs only in the server console or dev-mode output, never in the rendered HTML.

### F-02 — Two Simultaneous IN_PROGRESS WorkDays (Including a Future Date)
**Severity:** High  
**Area:** Scheduling / Data Integrity  
**Category:** Scheduling Logic / Data State  

**Steps to Reproduce:**
Query the `WorkDay` table on the live database.

**Actual DB State:**
| ID | Date       | Status      | Area                    |
|----|------------|-------------|-------------------------|
| 1  | 2026-03-23 | COMPLETE    | Edwinstowe (4-Weekly)   |
| 3  | 2026-03-28 | IN_PROGRESS | Edwinstowe (4-Weekly)   |
| 4  | 2026-04-25 | IN_PROGRESS | Edwinstowe (4-Weekly)   |

**Expected Result:**
- At most one WorkDay per area should be IN_PROGRESS at any time.
- A future WorkDay auto-created from a completed run should start as PLANNED (matches the schema default `@default(PLANNED)`).

**Actual Result:**
WorkDay 4 (April 25, 2026) is a future date with status `IN_PROGRESS`. WorkDay 3 (March 28, yesterday) is also `IN_PROGRESS`. Both reference the same area.

**Risk / Impact:**
- The Schedule page (`/days`) shows both runs as "Active," showing a pulsing clock icon for each. Workers could be confused about what to work on.
- The "Today" section on the schedule page checks for `date === today`. WD3 (March 28) would NOT appear in "Today" since it's yesterday — workers may not notice the overdue open run.
- WD4's pre-populated 12 PENDING jobs are "live" from the system's perspective, but the run isn't supposed to start until April 25.
- Can cause data confusion when WD3 is eventually completed: the `completeWorkDay` function will attempt to create the next run for `March 28 + 28 days = April 25` — but WD4 for April 25 already exists. The upsert would find it and not recreate it, but WD4 is already IN_PROGRESS (not PLANNED), which may cause unexpected behavior in the completion flow's status checks.

**Suggested Fix:**
1. When `completeWorkDay` creates the next auto-scheduled WorkDay, explicitly set `status: "PLANNED"` in the `create` clause of the upsert, rather than relying on the schema default.
2. Add a uniqueness or state check: prevent setting a WorkDay to IN_PROGRESS if another WorkDay for the same area is already IN_PROGRESS.
3. Consider adding a database constraint or application-level guard: only one non-COMPLETE WorkDay per area at a time.

### F-03 — Schedule View Has No "Overdue" Section for Past IN_PROGRESS WorkDays
**Severity:** High  
**Area:** Mobile Schedule Workflow  
**Category:** Product Gap / Scheduling Requirement  

**Steps to Reproduce:**
1. Sign in as an OWNER with tenant 1.
2. Navigate to `/days` (the Schedule page, primary mobile view).
3. Note that WD3 (March 28, IN_PROGRESS) is from yesterday.

**Expected Result:**
Per stated requirements: *"Mobile users should be able to rely primarily on the schedule view to see work due today or overdue."* An overdue run (IN_PROGRESS but past the scheduled date) should be prominently shown, distinct from upcoming or today's work.

**Actual Result:**
The schedule page's "Today" section only filters for days matching today's date (`isoDate(day.date) === todayISO`). WD3 (dated March 28, yesterday) is not today, so it appears only in the Calendar section, without any visual emphasis or "Overdue" label.

The `SchedulePageClient` component has these sections:
- **Today** — date-match only (misses yesterday's open run)
- **Calendar** — all days, including overdue ones buried among future dates

**Risk / Impact:**
A worker who opens the app on March 29 would see "Nothing scheduled today" in the primary section. The overdue run from March 28 is only visible by scrolling to the calendar. This violates the stated mobile workflow requirement.

**Suggested Fix:**
Add an "Overdue" section above "Today" in `SchedulePageClient` that shows all days where:
- `status === 'IN_PROGRESS'` AND `date < today`

Display these with a red/amber badge (e.g., "Overdue") at the top of the list so workers cannot miss them. This is the highest-priority UX improvement for the mobile role.

### F-04 — `/auth/forgot-password` Is Not in PUBLIC_ROUTES (Cannot Be Reached by Logged-Out Users)
**Severity:** Medium  
**Area:** Authentication  
**Category:** Usability / Logic Bug  

**Steps to Reproduce:**
1. Without being signed in, visit `https://wyndos.io/auth/forgot-password`.

**Expected Result:**
The forgot-password form should load (this is a public route by nature).

**Actual Result:**
The middleware redirects to `/auth/signin?callbackUrl=/auth/forgot-password`. After "sign in," the user would be sent back to `/auth/forgot-password` — but they cannot sign in because they have forgotten their password. This creates a circular redirect for locked-out users.

**Verified by:**
Live HTTP test: `https://wyndos.io/auth/forgot-password` returns 307 → `/auth/signin?callbackUrl=/auth/forgot-password`.

**Root Cause:**
`PUBLIC_ROUTES` in `middleware.ts` only contains `/auth/signin` and `/auth/signup`. The `/auth/forgot-password` path was not added.

**Suggested Fix:**
Add `/auth/forgot-password` to `PUBLIC_ROUTES` in `middleware.ts` (and any other auth-support routes such as `/auth/reset-password`).

*Note: The password reset flow itself is excluded from this test scope, but the route access block was confirmed through HTTP testing.*

### F-06 — Missing HTTP Security Headers
**Severity:** Medium  
**Area:** Infrastructure / Security  
**Category:** Security (OWASP A05 Security Misconfiguration)  

**Evidence (live production headers):**

| Header                       | Status    | Notes                                      |
|------------------------------|-----------|--------------------------------------------|
| `X-Frame-Options`            | MISSING   | Clickjacking risk                          |
| `Content-Security-Policy`    | MISSING   | XSS, injection risk                        |
| `Strict-Transport-Security`  | MISSING   | Should enforce HTTPS for all future visits |
| `X-Content-Type-Options`     | MISSING   | MIME type sniffing risk                    |
| `Referrer-Policy`            | MISSING   | Leaks referrer to third parties            |
| `Permissions-Policy`         | MISSING   | No feature restriction declared            |
| `Server`                     | PRESENT   | Exposes `nginx/1.18.0 (Ubuntu)` version    |

**Risk / Impact:**
- Without `X-Frame-Options` or CSP `frame-ancestors`, the application can be embedded in a third-party iframe (clickjacking).
- Without HSTS, browsers won't enforce HTTPS on subsequent visits if the HTTPS redirect fails.
- The `Server` header reveals the exact nginx version and OS, aiding targeted exploits.

**Suggested Fix:**
**Option A (preferred) — Add headers in `next.config.ts`:**
```ts
// next.config.ts
const nextConfig: NextConfig = {
  async headers() {
    return [
      {
        source: '/(.*)',
        headers: [
          { key: 'X-Frame-Options', value: 'DENY' },
          { key: 'X-Content-Type-Options', value: 'nosniff' },
          { key: 'Referrer-Policy', value: 'strict-origin-when-cross-origin' },
          { key: 'Permissions-Policy', value: 'camera=(), microphone=(), geolocation=()' },
          { key: 'Strict-Transport-Security', value: 'max-age=63072000; includeSubDomains; preload' },
        ],
      },
    ];
  },
};
```

**Option B — Add `server_tokens off;` and `add_header` directives to nginx config** (`/etc/nginx/sites-enabled/wyndos.conf`):
```nginx
server_tokens off;
add_header X-Frame-Options "DENY" always;
add_header X-Content-Type-Options "nosniff" always;
add_header Referrer-Policy "strict-origin-when-cross-origin" always;
add_header Strict-Transport-Security "max-age=63072000; includeSubDomains; preload" always;
```

### F-07 — API Endpoints Return 307 HTML Redirect for Unauthenticated Calls (Not 401 JSON)
**Severity:** Low  
**Area:** API  
**Category:** API Design / Developer Experience  

**Observed Behavior:**
Calling any protected API endpoint without auth (e.g., `/api/work-days`, `/api/areas`, `/api/customers/search`) returns `307 Temporary Redirect` to `/auth/signin?callbackUrl=...`, delivering an HTML sign-in page rather than a JSON error response.

**Risk / Impact:**
Any external client or automation calling the API would receive an HTML page instead of a JSON error. This would cause JSON parse failures in non-browser API consumers (e.g., future mobile apps, scripts, integrations).

**Suggested Fix:**
In the middleware, detect API routes and return `{ error: "Unauthorized" }` with 401 instead of redirecting:
```ts
if (pathname.startsWith("/api/") && !isLoggedIn && !pathname.startsWith("/api/auth") && pathname !== "/api/health") {
  return NextResponse.json({ error: "Unauthorized" }, { status: 401 });
}
```

### F-08 — No Rate Limiting on Login or Registration Endpoints
**Severity:** Medium  
**Area:** Authentication  
**Category:** Security (OWASP A07 Authentication Failures)  

**Observation:**
The application has no visible rate limiting on the credentials login endpoint (`/api/auth/callback/credentials`) or the registration Server Action. Multiple sequential login attempts with invalid credentials all received `302 → CredentialsSignin` without any slowdown, lockout, or CAPTCHA.

**Risk / Impact:**
- Credential stuffing and brute-force attacks are possible, limited only by network speed and server capacity.
- An attacker can automatically test large lists of email/password combinations.

**Suggested Fix:**
- Implement per-IP rate limiting at the nginx level (e.g., `limit_req_zone`), or
- Add application-level rate limiting using a library such as `@upstash/ratelimit` or similar, keyed by IP or email.
- Consider account lockout after N failed attempts within a time window (with unlock via email).

### F-10 — Service Had 7,000+ Restart-Loop Cycles (Historical)
**Severity:** Medium (Historical — now resolved)  
**Area:** Infrastructure / Deployment  
**Category:** Availability / Operational Risk  

**Evidence:**
```
Mar 27 23:53:xx systemd[1]: wyndos.service: Scheduled restart job, restart counter is at 7035–7046
```

**Observation:**
Before the SMTP display name fix on March 27, the service entered a restart loop cycling approximately every 5 seconds. The restart counter reached >7,000 before the service was manually stopped and fixed. The `wyndos.service` systemd unit does not have `StartLimitBurst` or `StartLimitIntervalSec` configured, meaning it will restart indefinitely if the application fails at startup.

**Risk / Impact:**
- Consuming unnecessary server resources (CPU, I/O) during a restart storm.
- No automatic alert is generated when the service enters a restart loop.
- If a similar startup failure occurs in the future (e.g., missing env var, DB connection failure), the service will restart indefinitely until manually identified and fixed.

**Suggested Fix:**
Add restart limit to `/etc/systemd/system/wyndos.service`:
```ini
[Service]
Restart=always
RestartSec=5
StartLimitBurst=10
StartLimitIntervalSec=120
```
This stops the restart loop after 10 failures in 2 minutes and lets the operator know the service needs attention.

Additionally, consider adding an alerting mechanism (e.g., email or webhook) when the service enters a failed state.

### F-12 — WorkDay Auto-Created by `completeWorkDay` Does Not Explicitly Set PLANNED Status
**Severity:** Low  
**Area:** Scheduling — Data Integrity  
**Category:** Code Quality  

**Evidence (actions.ts ~line 1797):**
```ts
const nextWorkDay = await prisma.workDay.upsert({
  where: { tenantId_date_areaId: { date: nextDue, areaId: workDay.area.id, tenantId } },
```

**Observation:**
The `create` clause does not specify `status`. The Prisma schema default is `PLANNED`, which should be correct. However, the current DB shows the auto-created WD4 as `IN_PROGRESS`, not `PLANNED`. This suggests either:

b) The upsert found an existing record (previously created via `scheduleAreaRun`) and did not update its status.

### F-13 — `scheduleAreaRun` Does Not Guard Against Scheduling Over an Already-IN_PROGRESS Day
**Severity:** Low  
**Area:** Scheduling  
**Category:** Logic / Data Integrity  

**Observation:**
The `scheduleAreaRun` function creates or finds a WorkDay for the given date and area, then sets the area's `nextDueDate` to that date. There is no check that prevents initiating a new run for an area that already has a run IN_PROGRESS. This means a user could:
1. Open Run A (WD3, March 28, IN_PROGRESS)
2. Schedule another run for the same area on April 25 (WD4)
3. Both are now IN_PROGRESS simultaneously

**Risk / Impact:**
Scheduling confusion; two open runs for the same area would confuse workers following the schedule view.

**Suggested Fix:**
Before creating a new WorkDay via `scheduleAreaRun`, check whether an IN_PROGRESS WorkDay already exists for that area:
```ts
const active = await prisma.workDay.findFirst({ where: { tenantId, areaId, status: 'IN_PROGRESS' } });
if (active) throw new Error(`${area.name} already has an active run. Complete it before scheduling a new one.`);
```

## 3. Unclear Behavior

### UB-01 — What Triggers a WorkDay Transition from PLANNED to IN_PROGRESS?
The `WorkDay` schema has three statuses: `PLANNED`, `IN_PROGRESS`, `COMPLETE`. The auto-created next-run WorkDay should be `PLANNED`. When does it transition to `IN_PROGRESS`? Looking at the code, `scheduleAreaRun` creates with no explicit status (defaults to `PLANNED`). There is no explicit "open day" or "start run" action visible in the source. Yet WD3 and WD4 are `IN_PROGRESS`. This transition mechanism is unclear from the codebase alone. It's possible the status changes to `IN_PROGRESS` the first time the day is viewed (on-load server action?), or when the first job is completed.

**Why This Matters:** If viewing a future day accidentally sets it to `IN_PROGRESS`, workers who preview the April 25 run would inadvertently activate it.

---

### UB-02 — WorkDay ID Gap (IDs 1, 3, 4 — ID 2 Missing)
The WorkDay sequence has a gap: IDs 1, 3, 4 exist; ID 2 does not. This suggests a WorkDay was created and then deleted, or the sequence was incremented without a corresponding record. No deletion mechanism for WorkDays is visible in the codebase (only `removeJob` is implemented).

**Why This Matters:** If WorkDays can be deleted, there is no audit trail of deletions. If ID 2 was never created (sequence skip), it suggests a partial transaction or rollback that deserves investigation.

---

### UB-03 — PLANNED WorkDay Status May Not Be Used Meaningfully in UI
The `SchedulePageClient` and `DayCard` components check for `status === 'COMPLETE'` and `status === 'IN_PROGRESS'` but do not render a distinct state for `PLANNED`. A `PLANNED` day would render as a plain white card with just a job count, not distinguishable from a calendar placeholder. If `PLANNED` days are meant to represent "upcoming but not started," their visual representation and accessibility for mobile workers is unclear.

---

### UB-04 — `nextRunAfter` Does Not Preserve Day-of-Week for Weekly Schedules
The code comment states "nextRunAfter preserves day-of-week for weekly schedules (addWeeks keeps weekday)," but the implementation uses `addUtcDays(fromDate, frequencyWeeks * 7)`. This correctly calculates the same weekday only if `frequencyWeeks` is an integer number of weeks — which it always is. However, if a Monday run is completed late on Friday, the NEXT run will be scheduled for Friday + 28 days (also a Friday), not the following Monday. Over time, the day of the run drifts from Monday to Friday. This may be intentional by design (completing when you can sets the new cadence) but could surprise operators who expect area runs on fixed weekdays.

## 4. Improvement Opportunities

### IO-01 — Add "Overdue Runs" Section to Schedule Page (Critical Mobile Improvement)
This is distinct from F-03 above but re-emphasised here. The mobile schedule workflow is the primary way workers interact with the system. Overdue work (past-date IN_PROGRESS days) must surface prominently — ideally at the top of the list with a "Overdue" warning, styled in amber/red.

---

### IO-02 — Show "Days Since Last Run" on Scheduler Card for Unscheduled Areas
Of the 21 areas, 20 have NULL scheduling dates. On the scheduler, these areas show no timing context. Adding "Never run" or "No history" indicators, alongside a prompt to schedule a first run, would help operators prioritise which areas need attention most urgently.

---

### IO-03 — Onboarding Wizard for First Run Setup
New tenants (like tenant 2 which has 0 customers and 0 areas run) would benefit from a guided onboarding step that prompts: "You have areas with no scheduled runs. Set them up now?" Reducing the friction to schedule the first run for each area would improve activation.

---

### IO-04 — Gracefully Hide or Disable Google Sign-In When Not Configured
Rather than the current error text, show nothing (or a disabled button with a tooltip in dev mode only). In production, if Google OAuth is not configured, the button and all related text should be completely hidden.

---

### IO-05 — Add Deployment Health Check to systemd
The current nginx → Node.js proxy returns responses immediately after Node.js starts, but the Next.js app may take seconds to be fully ready. Adding a health-check `ExecStartPost` that polls `/api/health` until it returns 200 would prevent the service from appearing "running" before it accepts requests.

---

### IO-06 — Customer Skip Flag Lacks Audit Trail
When `Customer.skipNextAreaRun = true`, there is no record of why it was set, when, or which area run triggered it. For a business that audits job history, this could cause confusion ("why was this customer skipped?"). Consider adding a `skipReason` or logging skip events in the job notes field when the skip is applied.

---

### IO-07 — Schedule Page "Today" Section Should Include Days Still IN_PROGRESS from Prior Dates
A slight enhancement to F-03: even if not labelled "Overdue," days that are `IN_PROGRESS` and were scheduled before today should appear alongside today's days, perhaps with a "Carry over from yesterday — still open" note.

---

### IO-08 — SUPER_ADMIN Support Session Expiry Is Not Enforced Server-Side
The `SUPPORT_SESSION_MAX_AGE_SECONDS = 7200` (2 hours) constant is set in the `switch-tenant` route and stored as a cookie, but no server-side expiry check validates that the session hasn't exceeded 2 hours. The cookie's max-age would expire client-side, but if the cookie is somehow extended or replayed, the server does not independently verify whether the support session is still within the time window. This should be validated against the `SupportAccessLog.createdAt` timestamp on every request that uses the support cookie.

---

### IO-09 — Area Schema Has `scheduleType` and `monthlyDay` Fields That Are Unused in the Current Scheduler UI
The `Area` model has `scheduleType: string` and `monthlyDay: integer | null` for MONTHLY scheduling support, but `scheduleType` is not visible or editable in the area management UI (could not be confirmed without a live browser session, but no UI code references for setting `scheduleType` were found). If monthly scheduling is intended, the UI should expose it; if not, the schema should be simplified to avoid confusion.

## 5. Blockers

### BL-01 — No Browser Automation / Real Session Available
All authenticated UI flows (customers page, day view, area management, payment entry, onboarding) require a live browser session with JavaScript execution. Without Playwright/Selenium or a pre-existing session token, authenticated page content cannot be fully tested via HTTP-only tooling. The registration Server Action specifically requires the Next.js client framework to encrypt and call the action, making it untestable via raw HTTP POST.

**Impact:** Cannot confirm: 
- Registration actually completes (Server Action not testable via curl)
- Full authenticated day view renders correctly
- Job completion flow and payment entry UI
- Mobile layout rendering on real devices

### BL-02 — No Worker Account to Test Worker-Specific Permissions
The live system has no WORKER-role user created for tenant 1. Worker permission testing (price visibility, scheduler access restriction) could not be exercised interactively.

### BL-03 — Tenant 2 Has Zero Customers and Zero Areas Run
Tenant 2 (hanna McMahon's) is empty — no customers, no workdays, no payments. Cross-tenant isolation testing was done through code review only (tenant ID always derived from JWT session, never from request params). An end-to-end cross-tenant access attempt (i.e., authenticated as tenant 2, attempt to access tenant 1 customer URL) could not be tested without live browser sessions.
